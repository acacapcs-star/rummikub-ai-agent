# Section 3：認知教練型 AI —— 從「打贏對手」到「陪伴引導」

> 個人延伸專案｜對應 `CognitiveHintEngine`

## 這一步做的是什麼、不是什麼

**這不是完整的「拉密抗失智 App」**，而是這個方向裡，第一個真正可以展示、可以編譯運作的核心元件：**漸進式提示引擎**。

之所以先做這個，是因為它是「教練型 AI」跟「競技型 AI」之間**最關鍵的技術分歧點**——`AIAgent_0`／`AIAgent_Conservative`／`AIAgent_Aggressive` 的目標都是「找出最好的一手，自己出牌」；`CognitiveHintEngine` 的目標是「找出對面玩家可行的一手，但不幫他出，只用由淺入深的方式提示」。

## 技術設計：沿用既有掃描邏輯，改變輸出目的

`findAttachableMove()` 的掃描邏輯，跟 `AIAgent_0::findRuns()` 裡「找出手牌能接到桌面 Run 頭尾」的概念完全相同——這是刻意的，因為兩者本來就是在回答同一個問題：「手牌裡有沒有牌能接到桌面上的某個 Run」。差別只在於：原本找到就直接呼叫 `applyProposedSets()` 出牌，這裡找到後改成組成一段自然語言提示。

三層提示，由淺入深：

| 層級 | 內容 | 設計邏輯 |
|---|---|---|
| `GENTLE_NUDGE` | 「手牌裡好像有牌可以出喔，仔細看看桌上的排列！」 | 只給信心，不給方向，讓玩家自己再想一次 |
| `POINT_TO_AREA` | 「留意一下桌上『紅色』那一排...」 | 縮小範圍到顏色/區域，但不講出具體是哪張牌 |
| `REVEAL_MOVE` | 「試試看把手上的『紅色 7』接到...」 | 直接講出答案 |

`tierFromStuckTurns()` 用「已經卡關幾回合」決定要給哪一層——回合數而不是真實時間，因為目前的引擎架構本身不追蹤時間，只追蹤回合，這點要誠實承認：真正的產品如果要做到「60 秒沒動作觸發提示」這種時間基準的設計，需要在更外層（例如 UI 層）加上計時器，不是這個模組本身能做到的。

## 誠實的限制

1. **掃描不到可行動作時，直接誠實回報「接不上」**，不會為了「一定要給出提示」而編造不存在的建議——這個設計選擇本身也是刻意的：寧可誠實地說「目前沒有」，也不要給錯誤的引導。
2. **三層的觸發門檻（卡關 2 回合／4 回合）是示意性數字，不是臨床驗證過的**——如果之後真的有機會找長輩實測，這裡的門檻很可能需要重新校準，甚至可能發現「回合數」本身就不是最適合的觸發依據，可能需要換成真實反應時間。
3. **目前只處理「接龍頭尾」這一種提示情境**，還沒涵蓋「破冰」「湊 Group」這些其他合法動作的提示——這是下一步可以擴充的方向。
4. **完全沒有做過任何長輩實測**，這一點在文件跟任何對外呈現的場合都要誠實講清楚，不能暗示已經驗證過成效。

## 下一步可能的方向

- 把這個模組接到 `HumanAgent` 的回合流程裡，讓「人類玩家卡關幾回合」變成真的可以觸發提示的即時功能，而不只是一個獨立函式庫
- 擴充涵蓋 Group 湊色、破冰分數不足這些其他情境的提示
- 如果有機會取得真實使用者（尤其是年長者）的測試機會，用實際反應時間取代「回合數」作為觸發依據

## 更新：完整串接並實測驗證（真實可玩的原型）

上面的內容是最初的設計規劃；這一段記錄後續實際把整套系統做完、串起來、實測驗證過的結果——**不是規劃，是真的跑起來看過的東西**。

### 完整串接的三層架構

1. **`CognitiveHintEngine`**：核心掃描與提示產生邏輯（本文件開頭描述的部分），後續再加入 `generateMeldHint()`，處理破冰前「手牌能不能湊到 30 分」這個獨立情境——原本只處理破冰後的接龍提示，這是刻意分開設計的，因為破冰前後玩家該關心的問題完全不同。
2. **`CognitiveCoachAgent`**：繼承 `HumanAgent`，每回合依 `initial_meld_done` 狀態自動切換呼叫 `generateHint()` 或 `generateMeldHint()`，並用桌面序列化字串比對來源持卡關回合數。
3. **`CoachHintBridge`**：一個刻意輕量的全域信箱＋回呼機制，讓 `CognitiveCoachAgent` 算出的提示能寫進 `GameManager` 匯出的 `state.json`，並透過回呼讓「這一輪」的匯出就反映最新提示，不用等到下一輪。

### 前端顯示

`visualizer/index.html` 加了一個獨立的提示欄位＋一小段自己輪詢 `state.json` 的 JavaScript，刻意不去動、不去猜測既有 `app.js` 的內部邏輯，把風險降到最低。

### 實測結果（真實截圖驗證，不是預期腳本）

**情境一：Turn 1，剛開局**
```
💡 手牌裡好像已經有一組不錯的牌可以試試看喔，仔細找找同色連續、或同數字不同色的組合！
```
`stuck_turns = 0`，正確給出最淺層的鼓勵性提示，不洩漏細節。

**情境二：Turn 5，同一組牌卡了幾輪之後**
```
💡 你手上有一組同色連續的牌，目前大約可以湊到 33 分，已經達標囉！
```
提示自動升級到第二層，而且「33 分」「已經達標」都是即時掃描手牌算出來的真實數字，不是預先寫死的文字——手牌裡藍色 10、11、12 剛好構成合法 Run，總分 33，系統正確判斷達標。

這證明了三件事都是真的在運作，不是表面看起來動、實際上是假的：卡關計數確實隨回合累積、提示層級確實依計數切換、分數確實是動態計算而非寫死。

### 過程中犯過的錯，誠實記錄

開發過程中，光是要把這個功能從「後端算得出提示」到「前端真的看得到提示」，中間踩了好幾個坑，值得記錄，因為每一個都反映一個真實的工程教訓：

- **前端抓取路徑猜錯兩次**：一開始猜測 `state.json` 的路徑跟頁面同層，後來才發現 `server.py` 實際把它放在 `/build/state.json`——這提醒了「不要用猜的，要去看伺服器實際的路由行為」。
- **最終真正的元兇：背景殘留的舊 `server.py` 程序**：前端程式碼、後端資料其實都已經是對的，但瀏覽器讀到的頁面卻一直是舊版——追了很久才發現，是有一個更早之前啟動、忘記關掉的 `server.py` 程序還占用著 port，新啟動的伺服器指令直接失敗（`Address already in use`），而系統一直在用那個舊程序回應請求。這個教訓很值得記下來：**當「檔案內容明明是對的，但畫面就是不對」時，要懷疑的不只是程式碼，也可能是環境裡有殘留的舊程序**。
