# 策略三：調牌與心機戰術 —— 讀牌、讀對手

> commit 順序 #3｜對應 PPT 第 8 頁「讀牌、讀對手，決定要不要出牌」

## 這是什麼

除了「怎麼打牌」，這個 agent 也會判斷「現在該不該打牌」。這裡包含兩個機制：一個是「囤牌」（自己藏招），一個是「反擊」（抓對手弱點）。

## 機制一：重疊牌型偵測（囤牌）

當手牌出現「同一個數字有 ≥2 種顏色、且鄰近數字也在手上」的情況時，代表這手牌同時有機會延伸成 Run 或 Group——這時候先按兵不動，冷卻 4 回合，保留之後的戰術彈性，而不是看到能出就出。

```cpp
bool internalCheckOverlap(const std::vector<Tile*>& hand) {
    // 同數字 >= 2 色，且該數字前後至少一個鄰居也在手上 → 觸發囤牌
}
```

但這個囤牌不是無條件的：牌堆剩餘 ≤20 張時會強制解除，避免因為囤牌反而讓自己在殘局吃虧。

## 機制二：對手抽牌監控（反擊）

追蹤牌堆剩餘張數 `draw_pile_size` 逐回合的變化。如果對手連續好幾輪都在抽牌（代表牌堆裡已經沒有他能用的組合），達到門檻後判定對手卡關，立刻觸發大風吹反擊，搶時間差多出牌。

```cpp
if (enemy_draw_count >= 8) {
    // 抓到卡牌空檔，立刻發動全域大風吹
}
```

這個門檻不是憑空定的——原本設定為 ≥3，但實測發現對手只要連續抽牌 3 次就幾乎每輪都觸發，反而干擾了正常的節奏，後來調高到 ≥8 才穩定下來，只在對手真正卡關時才出手。

## 想過但[尚未]做出來的構想：主動調牌／誘導對手

在開發過程中，其實有一個更進一步的想法：能不能刻意製造幾輪反常的出牌，觀察對手怎麼回應，藉此猜測對手是保守型還是衝動型的玩家，再用不同風格對付？

**誠實記錄一下結論：這個機制最終沒有寫成程式碼**，原因是引擎的 API 設計上就不允許——`playTurn()` 只給 `Board& board` 跟 `int draw_pile_size` 兩個參數，沒有任何管道可以得知「牌堆變化是哪一位對手造成的」，`Player::hand` 也是 `private`，沒有任何合法方式能看到對手手牌的蛛絲馬跡。

比較接近、而且引擎架構下真正做得到的版本，是機制二的「對手抽牌監控」——雖然看不到「是誰」抽牌、也看不到對手手牌內容，但可以用「牌堆總數變化」這個**公開資訊**，去推論「整體牌局是不是卡住了」，藉此決定要不要主動出手。這是在「不能偷看」的規則限制下，能做到最接近的近似版本。

## 延伸思考：4 人局的挑戰

如果對手變成 3 個人，`enemy_draw_count` 的語意需要從「盯著唯一那個對手」改成「觀察整桌流動性」——因為 3 個對手輪流動作時，你只看得到牌堆總數的變化，無法拆解出是哪一位對手造成的。這是這個專案下一階段想繼續深化的方向（詳見個人延伸專案的對手行為建模構想）。
