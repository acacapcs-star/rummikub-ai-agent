策略四：橫向縱向掃描模式 —— 二維對照表與顏色分堆的取捨


commit 順序 #4｜對應 PPT 第 3 頁「遊戲引擎怎麼運作」、internalCheckOverlap / findRuns / findGroups



這是什麼

拉密的合法牌組只有兩種：**Run（順組）**跟 Group（群組）。這兩種組合方式，本質上是對同一副牌「兩種不同方向」的檢視：

橫向：固定顏色，沿著數字軸看連續段 → 對應 Run
  紅色： 1 - 2 - 3 - 4 - 5 - 6 - 7 ...

縱向：固定數字，沿著顏色軸看是否湊滿 → 對應 Group
  數字 7： 紅7 - 藍7 - 黃7 - 黑7

這個「顏色 × 數字」的二維概念，最初是公民老師教的「二維對照表」做法：橫列是顏色、直行是數字，數字再由小到大排序，Joker 另外存放。核心概念完全一致，但實際寫進 ai_agent_0.cpp 時，同一個概念在不同地方用了兩種不同的資料結構，原因跟各自要解決的問題有關。

版本一：二維對照表（用在「是非題」）

internalCheckOverlap() 判斷「手牌有沒有重疊牌型」時，直接採用二維表：

cppbool internalCheckOverlap(const std::vector<Tile*>& hand) {
    std::vector<std::vector<int>> table(4, std::vector<int>(14, 0));
    // table[顏色][數字] = 1 或 0，就是「這個顏色+這個數字的牌，手上有沒有」
    for (Tile* t : hand) {
        if (!t || t->isJoker()) continue;
        table[static_cast<int>(t->getColor())][t->getNumber()] = 1;
    }
    // 之後只要查表格裡的 0/1，就能判斷有沒有重疊
}

這裡只需要回答「有沒有」，二維表是最合適的工具——查詢是 O(1) 直接定位，不需要額外掃描。

版本二：顏色分堆＋牌指標（用在「要動手組牌」）

但在 findRuns() 跟大風吹的掃描裡，用的是不同的結構：

cppstd::vector<std::vector<Tile*>> color_buckets(4);
// 每一堆裡放的是「真正的 Tile 指標」，不是 0/1

這裡不能只存「有沒有」，因為最後要把真正的牌排序、找出連續段、組成合法的 Run 送進 applyProposedSets()——二維表的 0/1 沒辦法拿去操作、排序、串接成真正能送出去的牌組。

兩者的取捨：沒有誰比較好，是解決不同問題

二維對照表顏色分堆＋牌指標適合回答「有沒有」「是不是重疊」這種是非題「幫我組出真正能出的牌」查詢效率O(1) 直接定位，更快需要額外排序、掃描連續段能不能組牌送出不行，只有 0/1，沒有牌本身可以，存的就是真正的 Tile*

如果只需要判斷「有沒有重疊」，二維表更精簡、更快；但只要牽涉到「要把牌實際組出來、排序、送出去」，就必須換成能操作真實物件的結構，二維表在這個情境下反而不夠用——這也是為什麼同一個「顏色×數字」的概念，在 ai_agent_0.cpp 裡會用兩種不同的實作方式：依照各自要解決的問題選擇合適的資料結構，不是刻意標新立異、也不是誰比誰更高明。
