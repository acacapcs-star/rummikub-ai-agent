# 策略二：大風吹 —— 全域重組演算法

> commit 順序 #2｜對應 PPT 第 6 頁「大風吹全域重組」

## 這是什麼

「大風吹」是這個專案投入最多心力、也是核心創新的部分（對應 `tryExtendBoard()`）。跟「一條龍」只看手牌不同，這裡是把桌面上已經打出去的牌、加上自己的手牌，全部收進同一個池子重新分堆掃描，找出比原本更長、更有效率的組合方式。

這是**一次性**的動作：收牌 → 分堆 → 保留 Group → 掃描 → 切點優化 → 驗證 → 送出，不是反覆丟回去、反覆衡量的迴圈。真正會「反覆呼叫」大風吹的地方是**殘局分支**——牌堆見底後迴圈呼叫這個函式直到打不出牌為止，但那是另一個獨立的機制，不是大風吹本身內建迴圈。

![大風吹全域重組流程圖](../../大風吹_flowchart.png)

## 五個步驟

**STEP 1／收牌入池**：桌面每個 set、加上自己的手牌，全部收進同一個池子。

**STEP 2／依顏色分堆（橫向掃描）**：依紅／藍／黃／黑分成四堆，Joker 另外集中存放。這一步只做橫向（同色）分堆，不做縱向（同數字找 Group）——縱向掃描 `findGroups()` 只有在**破冰階段**會用到，大風吹重組時不會重新去找新的 Group。

**STEP 3／保留合法 Group**：桌面上原本就合法的 Group，整組跳過、不拆進顏色堆。這一步是必要的保護機制——如果沒有這步，Group 被丟進「依顏色分堆」的邏輯裡，它的跨色分組資訊就會永久遺失，之後怎麼樣都拼不回去。

**STEP 4／掃描最長 Run**：每個顏色堆依數字排序後掃描找連續段，缺口張數 ≤ 剩餘 Joker 數時動態借 Joker 補上。

**STEP 5／長龍切斷**：當一條 Run 長度達到 6 張以上時，模擬每一個可能的切點，計算「切完之後手牌能接上哪一段」，選擇手牌能接最多張的切點，把長龍切成兩段。效果是：原本一條龍只有頭尾 2 個接入點，切成兩段之後變成 4 個接入點——在減少手上卡牌張數的同時，也增加了可以重組、切入的節點數量。

## 安全回滾鎖（Safety Rollback Lock）

重組完成、正式送出前，會逐一檢查：

1. 原桌面的每一張牌，是不是都還完整保留在新的合法組合裡？
2. 桌面的總張數，是不是確實比重組前增加了？

**只要有一項不符合，整批提案直接 rollback，回到重組前的狀態，絕對不會把不合法的盤面送進引擎。**

```cpp
if (all_board_tiles_safe && final_count > board_tiles_pool.size()) {
    board.applyProposedSets(this, final_legal_board);
    return total_tiles_played;
}
// 任何一絲不完美，強制回滾
board.applyProposedSets(this, safe_after_basic_extend_sets);
```

這個鎖本身是**被動的**——它只會「攔下」錯誤，不會「修正」錯誤。真正決定重組會不會成功的，是 STEP 2～4 的掃描邏輯本身夠不夠正確、夠不夠完整。

## 想過但沒做出來的構想：用「Joker 稀缺度」推斷對手手牌

開發過程中曾經想過：能不能觀察「連續好幾十局桌面上都沒出現 Joker、自己也沒抽到 Joker」，藉此推斷對手手上可能握有 Joker、進而調整策略的嚴謹程度？

**誠實記錄一下：這個機制最終沒有寫進 `ai_agent_0.cpp`**——目前的程式碼裡沒有任何「追蹤已經過了幾局」或「桌面連續幾局沒見到 Joker」的計數器。Joker 的實際處理分散在三個地方：破冰階段的排序考量（用量少者優先）、大風吹掃描時動態填補缺口、殘局階段把剩餘 Joker 塞進 3 張的組合裡消分。這三個都是**規則層面**的處理，跟「觀察局數、推斷對手是否持有 Joker」這種**對手建模**是完全不同層次的東西，後者目前還停留在構想階段。

## 原本規劃 vs 最終實作

開發過程中原本規劃用遞迴窮舉（`solveRummikub`）來找真正的最佳解，但考量到需要疊加安全回滾鎖來確保正確性，最後改用「迭代＋貪心規則」整套寫進 `tryExtendBoard`。`solveRummikub` 保留了函式介面，但目前未實際使用——這是留給未來的優化方向：把貪心啟發式換成真正的遞迴／動態規劃搜尋。
